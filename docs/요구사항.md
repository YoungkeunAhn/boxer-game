# 복서키우기 앱인토스 출시 관련 문서화 프롬프트

너는 게임 출시 기획자, 테크니컬 라이터, 웹 프론트엔드 아키텍트 역할을 함께 수행한다.

나는 `복서키우기`라는 방치형 복싱 게임을 만들고 있다.
이 게임은 일반 웹게임이 아니라, 추후 `토스 앱인토스 미니앱` 출시를 목표로 한다.

이미 아래 문서는 작성되어 있다.

* `docs/platform-apps-in-toss.md`

따라서 이번 작업에서는 아래 3개 문서만 새로 작성해줘.

* `docs/release-checklist.md`
* `docs/game-rating.md`
* `docs/technical-stack.md`

## 공통 작성 규칙

1. 각 문서는 최대 100줄 이내로 작성한다.
2. 문서 내용은 한국어로 작성한다.
3. 파일명은 영어로 유지한다.
4. Markdown 형식으로 작성한다.
5. 문서는 실무에서 바로 저장해서 사용할 수 있는 수준으로 작성한다.
6. 너무 추상적으로 쓰지 말고, 실제 개발/출시 준비에 바로 참고할 수 있게 작성한다.
7. 아직 확정되지 않은 내용은 `TODO`로 표시한다.
8. 외부 정책이나 앱인토스 기준이 필요한 항목은 `공식 문서 확인 필요`라고 표시한다.
9. 토스 앱인토스 정책, 게임 심의, 등급분류, SDK 요구사항은 변경될 수 있으므로 단정하지 않는다.
10. 이미 작성된 `docs/platform-apps-in-toss.md`와 내용이 중복되지 않게 작성한다.
11. 각 문서 하단에는 관련 문서 링크를 넣는다.
12. README.md 전체를 다시 작성하지 말고, README에 추가할 문서 링크 섹션만 별도로 제안한다.

## 프로젝트 기본 정보

프로젝트명: 복서키우기

장르:

* 방치형 성장 게임
* 복싱 육성 게임
* 웹 기반 미니게임

출시 목표:

* 토스 앱인토스 미니앱 출시
* 초기 버전은 WebView 기반 미니앱으로 개발
* Unity, Godot 같은 게임 엔진은 초기 버전에서 사용하지 않음

기술 방향:

* React
* Vite
* TypeScript
* Zustand
* localStorage 또는 추후 서버 저장
* 앱인토스 WebView SDK 연동 예정

게임 핵심 루프:
훈련 → 능력치 상승 → 경기 → 보상 획득 → 업그레이드 → 더 강한 상대 도전

v0.1 MVP 기능:

* 복서 생성
* 복서 상태 보기
* 훈련하기
* 상대와 경기하기
* 경기 결과 계산
* 보상 지급
* 저장/불러오기
* 재접속 시 진행도 유지

v0.1 제외 기능:

* 인앱 결제
* 인앱 광고
* 리더보드
* 친구 초대
* PVP
* 길드
* 시즌패스
* 복잡한 스킬 시스템

# 작성할 문서 1

## FILE: docs/release-checklist.md

역할:

* 복서키우기를 앱인토스에 출시하기 전 확인해야 할 작업 목록을 정리한다.
* 개발, QA, 검수, 제출 준비를 한 문서에서 확인할 수 있게 한다.

포함할 내용:

* 문서 목적
* 출시 단계 요약
* 개발 완료 체크리스트
* 앱인토스 연동 체크리스트
* 게임 저장/불러오기 체크리스트
* QA 체크리스트
* 리소스 준비 체크리스트
* 검수 제출 전 체크리스트
* v0.1에서는 제외할 항목
* 출시 전 최종 확인 항목

반드시 포함할 체크리스트 예시:

* 게임이 앱 안에서 정상 실행되는가
* 새로고침 또는 재접속 후 데이터가 유지되는가
* 게임 진행이 막히는 구간이 없는가
* 사운드가 있다면 On/Off 처리가 되는가
* 종료 또는 이탈 상황에서 데이터 손실이 없는가
* 앱 아이콘, 썸네일, 설명 문구가 준비되었는가
* 고객 문의 이메일 또는 문의 경로가 준비되었는가
* 공식 앱인토스 검수 기준 확인이 완료되었는가

주의:

* 법적/정책적 내용은 단정하지 말고 `공식 문서 확인 필요`라고 표시한다.
* 실행 가능한 체크리스트 형태로 작성한다.
* 각 항목은 체크박스 Markdown 형식을 사용한다.

# 작성할 문서 2

## FILE: docs/game-rating.md

역할:

* 복서키우기가 게임으로 분류될 가능성이 있으므로, 게임 등급분류 준비 항목을 정리한다.
* 실제 심의 또는 자체등급분류 신청 전에 확인해야 할 내용을 정리한다.

포함할 내용:

* 문서 목적
* 왜 등급분류 검토가 필요한지
* 확인해야 할 등급분류 경로
* 복서키우기의 콘텐츠 리스크 점검
* 폭력성 관련 검토
* 결제/광고/랜덤박스 관련 검토
* 개인정보/계정 저장 관련 검토
* v0.1 기준 안전한 콘텐츠 방향
* 준비해야 할 자료
* TODO 목록

주의:

* 법률 자문처럼 단정하지 않는다.
* `게임물관리위원회`, `앱스토어 자체등급분류`, `플레이스토어 자체등급분류`, `원스토어 자체등급분류` 같은 경로는 검토 대상으로만 언급한다.
* 실제 제출 가능 여부나 조건은 `공식 문서 확인 필요`라고 작성한다.
* 복서키우기의 폭력성은 복싱 경기 표현이므로, 표현 수위를 낮추는 방향을 제안한다.
* 피, 잔혹한 연출, 도박성 랜덤 보상, 유료 확률형 아이템은 v0.1에서 제외하는 방향으로 작성한다.

포함할 내용 예시:

* 복싱은 격투 스포츠이므로 폭력성 표현 수위 검토 필요
* v0.1에서는 캐릭터 타격 애니메이션을 과격하게 만들지 않음
* 피격, 출혈, 부상, 잔혹 표현은 제외
* 랜덤박스, 확률형 유료 아이템은 제외
* 광고/결제는 v0.1에서 제외
* 등급분류 자료로 게임 설명, 스크린샷, 플레이 영상, 기능 목록을 준비

# 작성할 문서 3

## FILE: docs/technical-stack.md

역할:

* 복서키우기의 초기 기술 스택과 구조를 정의한다.
* 개발자가 어떤 방식으로 프로젝트를 시작해야 하는지 알 수 있게 한다.

포함할 내용:

* 문서 목적
* 기술 스택 요약
* 왜 Unity가 아니라 WebView 기반 웹앱으로 시작하는지
* 프론트엔드 기술 스택
* 상태 관리 방식
* 저장 방식
* 게임 로직 분리 원칙
* 추천 폴더 구조
* v0.1에서 사용하지 않을 기술
* 추후 확장 가능성

기술 스택은 아래 기준으로 작성:

* Vite
* React
* TypeScript
* Zustand
* CSS Modules 또는 Tailwind CSS
* localStorage
* 앱인토스 WebView SDK
* 추후 서버 저장은 TODO 처리

추천 폴더 구조 예시를 포함:

```txt
src/
  game/
    types.ts
    constants.ts
    formulas.ts
    training.ts
    battle.ts
    save.ts
  data/
    trainings.ts
    opponents.ts
    equipments.ts
  stores/
    gameStore.ts
  components/
    BoxerStatus.tsx
    TrainingPanel.tsx
    BattlePanel.tsx
  pages/
    HomePage.tsx
```

반드시 포함할 원칙:

* 게임 로직은 UI와 분리한다.
* 전투력 계산, 훈련 처리, 경기 결과 계산은 순수 함수로 작성한다.
* 저장/불러오기는 별도 모듈로 분리한다.
* v0.1에서는 서버 저장보다 localStorage로 빠르게 검증한다.
* 앱인토스 사용자 식별자 기반 서버 저장은 추후 검토한다.
* SSR에 의존하지 않는 구조로 작성한다.
